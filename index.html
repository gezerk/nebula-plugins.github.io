Welcome to Nebula

The nebula-plugins organization was setup to facilitate the generation, governance, and releasing of Gradle plugins. It is done by providing a space to host plugins, in SCM, CI, and a Repository. A single GitHub organization is used, for which anyone or any plugin can be added. Cloudbees jobs are created for every plugin to provide a standard set of jobs. Releases are posted to bintray, proxied to JCenter and synced to Maven Central. While not all steps are automated (yet), the goal to a self-service fast controlled flow plugins out to entire Gradle community.

nebula-plugins was founded by Justin Ryan (@quidryan) as an effort at Netflix to share its internal build infrastructure.

SCM

The GitHub organization nebula-plugins (https://github.com/nebula-plugins/) is just that, an organization. Any willing contributor can be added to the organization. That allows a level of collaboration between interested parties that isn't available when a plugin is locked up in an user's account.

Continuous Integration

Using Cloudbees, there's a hosted Jenkins instance (via Netflix) to host build jobs (https://netflixoss.ci.cloudbees.com/job/nebula-plugins/). The Job DSL plugin is used to ensure the jobs are standardize, and updated when improvements are made (https://netflixoss.ci.cloudbees.com/job/nebula-plugins/job/SEED-nebula-plugins/). The core jobs are ones to build the HEAD, check pull requests, and release the plugin.

Repositories

Upon release, a plugin is uploaded to the nebula-plugins organization in bintray (https://bintray.com/nebula/gradle-plugins). If configured correctly, those uploads are linked to JCenter, meaning they're instantly available to any Gradle user using the jcenter() repository. They are then synced to Maven Central under the com.netflix.nebula group name.

Contributing

Every contribution is highly welcome whether that's documentation, bugfixes, new features or improved test coverage. Nebula follows the usual GitHub pull request model. When submitting a pull request, please open it against the current default branch of the project. The default branch indicates the compatible Gradle version. For example if the branch is named gradle-1.12 it means that it is developed and tested against Gradle version 1.12. After merging a pull request into the default branch the Nebula team will take care of merging the change to branches dedicated to later Gradle versions. Thanks for your contributions!
